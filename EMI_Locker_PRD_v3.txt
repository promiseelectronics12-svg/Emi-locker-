SECTION 1: PRE-DEVELOPMENT — CRITICAL QUESTIONS ANSWERED

Q1. Can the app take Device Owner permission and act as the owner of the phone?
YES. Android's Device Policy Controller (DPC) mechanism grants the app system-level ownership. It is not a hack. It is Google's official enterprise MDM framework used globally by Samsung Knox, Microsoft Intune, and all major MDM providers.

Q2. Can we block Developer Options and USB Debugging after becoming Device Owner?
YES — completely and permanently. Using DevicePolicyManager.setGlobalSetting(), the app disables ADB and hides Developer Options entirely. Even if a user connects the phone to a PC with special tools, ADB cannot connect because it is disabled at the Android system policy level. This cannot be re-enabled without Device Owner authorization.

Specific policies enforced immediately after enrollment:
- setGlobalSetting(ADB_ENABLED, 0) — USB debugging disabled
- setGlobalSetting(DEVELOPMENT_SETTINGS_ENABLED, 0) — Developer Options hidden
- setUsbDataSignalingEnabled(false) — USB data blocked, charging only
- setSecureSetting(INSTALL_NON_MARKET_APPS, 0) — sideloading blocked
- Battery optimization exempted so OEM power managers cannot kill the app

Q3. Can we use Google's own infrastructure to back our app?
YES. By registering as a Google EMM (Enterprise Mobility Management) partner and using Android Management API (AMAPI), our app is backed by Google's own enterprise device management infrastructure. Our app becomes a recognized authority in the Google ecosystem, not a third-party app fighting the OS.

Q4. The Managed Google Account — how do we handle it and will Google block us?
We register as a Google Workspace Enterprise EMM partner. For each enrolled device, AMAPI automatically provisions a managed Google account from our enterprise domain, for example device-uuid@emilocker-mdm.com. We never store raw Google account passwords in our database. AMAPI manages the account lifecycle. We only store the AMAPI device ID and enrollment token. Google will NOT block this. This is the exact intended use case of Android Enterprise and AMAPI. The managed account is configured as a work profile account. It cannot be used for YouTube, Gmail personal, or any consumer Google service. It exists only to enforce FRP. When a device is decoupled after full payment, we call the AMAPI API to delete the managed account. Google processes the deletion, the FRP clears, and the user can freely set up their own Google account.

Q5. Can we add the device using a QR code scanner from our APK?
YES. The Dealer App generates a QR code. During the phone's first boot setup wizard screen, the dealer taps the screen 6 times to trigger QR enrollment mode, then scans the QR code. Android reads it, downloads our app via AMAPI, installs it with Device Owner privileges automatically. The user never sees an install prompt.

The QR code contains:
- android.app.extra.PROVISIONING_DEVICE_ADMIN_COMPONENT_NAME — our app's DPC component
- android.app.extra.PROVISIONING_DEVICE_ADMIN_PACKAGE_DOWNLOAD_LOCATION — download URL
- android.app.extra.PROVISIONING_DEVICE_ADMIN_SIGNATURE_CHECKSUM — security hash
- android.app.extra.PROVISIONING_LOCALE — language setting
- Our custom enrollment token — ties this device to the specific EMI agreement and dealer

Q6. BTRC NEIR — can we integrate it?
BTRC launched the National Equipment Identity Register (NEIR) system which went live January 2026. It links every phone's IMEI to a specific NID and SIM card. Unregistered or flagged IMEIs are blocked from all Bangladesh telecom networks — no calls, no mobile data, no bKash, no Nagad.

BTRC NEIR does not currently have a public API for third-party fraud reporting. The NEIR portal handles registrations via email and web form, not API. Our NEIR strategy is three phases:
- Phase A: Add NEIR reporting clause to user consent form as a deterrent
- Phase B: When Admin confirms fraud, submit IMEI to BTRC via neir@btrc.gov.bd manually with NID evidence
- Phase C: Apply for API access when BTRC opens it for fintech and MDM partners

Since our dealers are registered mobile phone businesses, they are already required to register their device inventory in NEIR. Our Dealer App will have a NEIR Export feature that generates the required Excel file for BTRC submission, making our platform useful beyond EMI enforcement.

Q7. Dealer lock requests — how does server verification work?
Dealers CANNOT directly lock devices. They submit a lock REQUEST. The server acts as a neutral judge. The server verifies if the reason is valid based on the actual EMI schedule data. Invalid requests are rejected. This prevents dealers from abusing the lock feature for non-payment reasons.

Q8. Decoupling — final decision on the 5-day window?
CONFIRMED FINAL: The 5-day dealer verification window stays. However dealer approval is NOT required for decoupling and dealers CANNOT block decoupling after payment is confirmed. The dealer's role in the window is only to FLAG fraud. If no fraud flag is raised by day 5, Admin is notified and executes the decoupling. Dealer cannot block a paid-off customer.

SECTION 2: SYSTEM OVERVIEW AND ROLE HIERARCHY

The Four Actors:
Platform Owner (Us) — Controls everything including server, keys, and final decoupling. Uses Admin Web Panel. Can fully lock any device.
Reseller — Regional distributors we authorize. Activates dealers and sells activation keys. Uses Reseller App built in Flutter. Cannot lock devices.
Dealer — Mobile phone shops and NBFC agents. Enrolls devices and manages EMI. Uses Dealer App built in Flutter. Can only submit lock requests — server verifies.
User (EMI Holder) — Buys phone on EMI and uses phone within constraints. Uses User EMI App built in Kotlin. Cannot uninstall the app or modify MDM in any way.

The Three Apps:
User EMI App — Android only, built in Kotlin. Installed as Device Owner at point of sale. Cannot be deleted or stopped.
Dealer App — Android and iOS, built in Flutter. Manages device enrollment, EMI schedule, and submits lock requests.
Reseller App — Android and iOS, built in Flutter. Manages dealer accounts and key inventory.

Architecture decision: Dealer App and Reseller App share one Flutter codebase with role-based UI switching. One APK, two roles.

Server Infrastructure:
- Backend API: Node.js + Express — all business logic, auth, key management, EMI schedules
- Relational Database: PostgreSQL — accounts, keys, agreements, audit logs, security events
- Realtime Database: Firebase Realtime DB — device state, lock status, changes reach device in under 2 seconds
- Push Notifications: Firebase Cloud Messaging — all push to all three apps plus offline fallback triggers
- Device Management: Android Management API (AMAPI) — Google's MDM backbone
- Key Security: HMAC-SHA256 plus HSM — tamper-proof key signing and command signing
- Auth: JWT plus TOTP 2FA — all role logins require 2FA
- Hosting: Google Cloud Platform — same infrastructure as AMAPI for lowest latency

SECTION 3: SECURITY ARCHITECTURE — 8 LAYERS

Every layer is independent. Breaking one layer does not break the others.

Layer 1 — Device Owner Policy (enforced by our app via AMAPI): Blocks app deletion, ADB, Developer Options, permission revocation
Layer 2 — Factory Reset Protection (enforced by Google Infrastructure): Blocks post-reset phone use without our managed account credentials
Layer 3 — Managed Google Account (enforced by AMAPI and Google Workspace): Blocks FRP bypass — account is ours, not the user's
Layer 4 — Certificate Pinning (enforced by our app at network layer): Blocks man-in-the-middle attacks and fake server interception
Layer 5 — Command Signing via HSM (enforced by our server and app): Blocks fake or replayed lock and unlock commands
Layer 6 — Hardware Binding (enforced by our server and app): Blocks commands used on a different device — uses IMEI plus Serial plus SoC
Layer 7 — APK Integrity Check (enforced by our server): Blocks tampered or cloned versions of our app
Layer 8 — Samsung Knox Fuse (enforced by Samsung hardware — Samsung only): Blocks post-bootloader-unlock device use — permanent and irreversible

Device Owner Policies enforced immediately after enrollment:
- setGlobalSetting(ADB_ENABLED, 0)
- setGlobalSetting(DEVELOPMENT_SETTINGS_ENABLED, 0)
- setUsbDataSignalingEnabled(false)
- setSecureSetting(INSTALL_NON_MARKET_APPS, 0)
- addPersistentPreferredActivity() — our app cannot be replaced as Device Admin
- setUninstallBlocked(componentName, true) — uninstall button permanently grayed out
- setApplicationHidden(componentName, false) — app cannot be hidden
- DeviceAdminReceiver onDisableRequested() — any attempt to remove admin triggers re-lock

Command Signing — every command the server sends is signed with our HSM private key. Each command contains:
- Device IMEI — command is device-specific, cannot be replayed on another phone
- Timestamp — command expires in 5 minutes, prevents replay attacks
- Nonce — one-time random string, used nonces are rejected
- Action type — lock, partial_lock, unlock, decouple, message
- HSM Signature — signed with our private key, verified with embedded public key in app

Play Integrity Attestation — on every app launch and daily background check:
- Verifies device has not been rooted
- Verifies APK has not been tampered with
- If attestation fails: device is immediately flagged, dealer alerted, device locked automatically

Tamper & Hardware Monitoring (Real-time Alerts):
- SIM Swap Detection: App monitors for SIM card changes. If the ICCID changes, the app reads the new phone number (auto-granted READ_PHONE_STATE via DPC) and sends a HIGH priority alert to the dealer.
- USB / ADB Tamper Detection: App monitors USB connection states and Developer Options access attempts. If suspicious USB activity or ADB access attempts are detected, a TAMPER warning is immediately fired to the dealer. The dealer handles the business risk.

SECTION 5: LOCK REQUEST AND SERVER VERIFICATION LOGIC

Dealers CANNOT directly lock devices. They submit requests. The server is the judge.

Lock Request Flow:
1. Dealer opens a device in Dealer App and taps Request Lock
2. Dealer selects a lock reason from a predefined list
3. Dealer optionally adds a note of max 200 characters
4. Request is sent to server with dealer ID, device ID, reason code, timestamp, and dealer note
5. Server runs Verification Engine against current EMI data for this device
6. If VALID: server schedules the appropriate lock level and sends it to the device
7. If INVALID: server rejects. No lock is sent.

Valid Lock Reason Codes:
OVERDUE_1 — Payment 1 to 3 days overdue — Server approves Reminder mode only
OVERDUE_3 — Payment 3 to 7 days overdue — Server approves Partial Lock
OVERDUE_7 — Payment 7 or more days overdue — Server approves Full Lock
SUSPECTED_SALE — Dealer suspects device was sold or transferred — Escalates to Admin
SUSPECTED_FRAUD — Dealer suspects fraud or stolen claim — Escalates to Admin immediately

Server-Automated Locking schedule:
7 days before due — No lock — Reminder push notification only
3 days before due — No lock — Warning overlay push
Due date Day 0 — No lock — Full-screen overlay on unlock
Day 1 overdue — Reminder mode — Overlay every 30 min non-dismissible
Day 3 overdue — Partial Lock — Only calls, SMS, EMI app work
Day 7 overdue — Full Lock — Kiosk mode, only emergency and dealer contact
Day 14 overdue — Full Lock plus Flag — Admin notified, NEIR report considered

Technical Foundation:
- Language: Kotlin — native Android only. No Flutter, no React Native.
- Package Name Obfuscation: App must use `com.android.system.stk.services` to masquerade as SIM Toolkit.
- Battery Immunity: App must declare REQUEST_IGNORE_BATTERY_OPTIMIZATIONS and use Device Owner privileges to whitelist itself from OEM app killers.

- Minimum Android version: Android 8.0 (API 26)
- Device Owner provisioned via AMAPI QR code at first boot
- Persistent foreground service — DeviceAdminReceiver restarts service if killed
- FCM integration for push commands from server
- Play Integrity API called on every launch and daily background check
- Certificate pinning — only connects to api.emilocker.com

The Six Screen States:
1. Normal — Payment current — Full phone access
2. Reminder — 7 days before due — Push notification and app badge
3. Warning — 3 days before due — Persistent top banner dismissible for 4 hours
4. Overdue Alert — Day 1 after due — Full-screen overlay on every unlock
5. Partial Lock — Day 3 after due — Calls, SMS, EMI app, emergency calls only (setPackagesSuspended)
6. Full Lock — Day 7 after due — Kiosk mode, emergency + dealer contact only (lockNow + setLockTaskPackages)

Overlay Technical Implementation:
- Permission: SYSTEM_ALERT_WINDOW granted at Device Owner level, user cannot revoke
- Window type: TYPE_APPLICATION_OVERLAY
- In Partial Lock: setPackagesSuspended() suspends all apps except Phone, Messages, EMI app, Emergency dialer
- In Full Lock: lockNow() called plus setLockTaskPackages() restricts to EMI app only

Offline Resilience — Pre-Authorized Unlock Tokens (PAUT):
- Signed JWT containing: device IMEI, authorized unlock datetime, expiry of 48 hours, server HSM signature
- Stored encrypted in app's local storage
- If FCM unlock fails: app checks for valid PAUT at authorized time and unlocks locally
- On next internet connection: device reports PAUT-triggered unlock to server for audit log

Hardware & Security Monitoring Services:
- SimChangeReceiver: Listens for ACTION_SIM_STATE_CHANGED. Reports new SIM ICCID/Phone Number to backend -> Dealer Alert.
- UsbTamperReceiver: Listens for USB connection state and Security Logging events (if DPC logging enabled) to detect ADB/tamper attempts -> Dealer Alert.

SECTION 10: KEY MANAGEMENT SYSTEM

Key Format: XXXX-XXXX-XXXX-XXXX (e.g., A7K2-9PQR-3XNM-8WVT)
Character set: Uppercase A-Z plus digits 2 through 9, excluding O, 0, I, 1, S, 5
Entropy: approximately 77 bits — unguessable by brute force

Each key is: single-use, time-limited (72 hour expiry), dealer-bound, IMEI-bound at activation, HMAC-SHA256 signed

Key Lifecycle:
PENDING_ADMIN → APPROVED → GENERATED → ASSIGNED → CONSUMED (IMEI bound) / EXPIRED (72hr unused)

Anti-Fraud Controls:
- Reseller cannot request more than 20% of monthly quota in single request
- Dealer using more than 10 keys in 24 hours triggers admin review
- Key encoded with dealer ID — wrong dealer means invalid key
- Key is permanently burned to IMEI at enrollment

SECTION 11: DECOUPLING SYSTEM — FINAL LOGIC

FINAL DECISION: Payment verification is the ONLY decoupling trigger. Dealer approval is NOT required.

Decoupling State Machine:
EMI_ACTIVE → FINAL_PAYMENT_RECEIVED → DEALER_NOTIFIED → (fraud flag?) → PENDING_ADMIN_DECOUPLE → DEVICE_DECOUPLED

What Happens When Admin Decouples:
1. Admin clicks Execute Decoupling — requires 2FA confirmation
2. Server generates signed Decouple Command with RTOC (Release Token One-time Code)
3. Command sent to device via FCM
4. App executes: clears lock policies → removes Device Owner → deletes managed account via AMAPI → uninstalls itself
5. Device reboots as clean normal Android phone
6. Server marks device as DECOUPLED — audit log records timestamp, admin ID, RTOC used

If FCM Fails During Decoupling:
- Server issues Pre-Authorized Decouple Token (PADT) — expires in 7 days
- Device checks for pending PADT on every network reconnection
- Customer can visit dealer who pushes decouple via Dealer App

SECTION 12: OFFLINE RESILIENCE AND FCM FALLBACK

Three-Channel Command Delivery:
Channel 1 — FCM Push — Primary, 95%+ delivery
Channel 2 — AMAPI Policy Push — Secondary, very high reliability, bypasses FCM
Channel 3 — Local PAUT/PADT Token — Tertiary, 100% offline, no network needed
